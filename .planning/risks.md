### 위험 1️⃣: Google Maps API 학습 곡선 & 성능

**카테고리**: 기술 (Technology)  
**영향도**: 🔴 **높음** (일정 3-4일 지연 가능)  
**발생확률**: 🟡 **중간** (처음 써보므로)  
**총 위험도**: **높음**

#### 설명

Google Maps API는 Flutter에서 처음 사용해보는 라이브러리입니다:
- 지도 렌더링 성능 최적화 모름
- 마커 많을 때 성능 저하 가능
- iOS/Android 플랫폼별 차이 있음

#### 영향

```
최악의 경우:
- 지도 로딩 시간 > 5초 (목표: < 2초)
- 마커 50개 표시시 프레임 드롭
- 중간발표 데모 실패 가능
- 일정 3-4일 지연
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
11주차 초반:
□ Google Maps Flutter 튜토리얼 2-3개 완료
□ 공식 문서 읽기 (Clustering, Markers)
□ Sample 앱 만들기 (지도 표시만)

11주차 중반:
□ 성능 테스트 (10개, 50개, 100개 마커)
□ 최적화 기법 미리 학습 (GeoHash, Clustering)
□ iOS/Android 동시 테스트
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ 지도 로딩 시간 측정
□ 마커 렌더링 성능 확인
□ 프레임 드롭 없는지 확인
```

**응급 대응 (Contingency)**:
```
만약 지도 성능이 나쁜 경우:
1. 마커 클러스터링 적용 (1-2일)
2. 페이지네이션 적용 (반경 500m씩)
3. 그 외 지도 기능은 나중 (Should→Won't)
4. 최악: 지도 대신 리스트 뷰 (완전 재설계 아님)
```

**일정 버퍼**: **+4일** (11주차에 추가)

### 위험 2️⃣: Firebase 실시간 동기화 복잡도

**카테고리**: 기술 (Technology)  
**영향도**: 🟡 **중간** (일정 2일 지연)  
**발생확률**: 🟡 **중간** (처음 실시간 DB 사용)  
**총 위험도**: **중간**

#### 설명

실시간 데이터 동기화:
- Firestore Listeners 설정 미숙
- 데이터 일관성 문제 (로컬 ≠ 서버)
- 메모리 누수 (Listener 정리 안 함)
- 동시 예약 충돌

#### 영향

```
예약 시스템에서:
- 두 명이 동시에 같은 시간대 예약
- 한 명은 확인되지만 나머지는 취소됨
- 사용자 불만 & 데이터 오염
- 13주차에 발견되면 일정 3-5일 지연
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
12주차 예약 시스템 구현시:
□ Firestore 트랜잭션 학습 (conflict resolution)
□ Optimistic UI 업데이트 패턴 학습
□ 동시성 테스트 (2개 기기 동시 예약)
□ Listener cleanup 패턴 확립

테스트:
□ 단위 테스트 작성 (예약 생성 로직)
□ 동시성 테스트 (Firebase Emulator 사용)
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ 예약 데이터 일관성 확인
□ 메모리 누수 없는지 확인
□ 리스너 개수 모니터링
```

**응급 대응 (Contingency)**:
```
만약 동기화 문제 발생:
1. 로컬 캐싱 도입 (2일)
2. 낙관적 UI 업데이트 (1일)
3. 서버 재구조화 (3-4일)
4. 최악: 예약 → 찜하기로 변경 (기능 축소)
```

**일정 버퍼**: **+3일** (13주차에 추가)

### 위험 3️⃣: 시험 및 과제 폭주

**카테고리**: 일정 (Schedule)  
**영향도**: 🔴 **매우 높음** (프로젝트 일시 중단 가능)  
**발생확률**: 🟡 **중간** (12월은 평상시 바쁨)  
**총 위험도**: **높음**

#### 설명

대학 학사일정:
- 11월 말: 기말고사 (프로젝트 1주 정지)
- 12월: 기말 과제/프로젝트 폭주
- 12월 초~중: 다른 과목 중간/마지막 과제

#### 영향

```
최악의 경우:
- 11월 말: 1주일 진행 불가 (고사 준비)
- 12월 초: 3-4일 진행 불가 (다른 과제)
- 12월 중: 불규칙한 진행 (여러 deadline)
- 결과: 총 8-10일 손실 = 일정 큰 지연
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
10-11주차:
□ 학사 일정 달력 작성
□ 다른 과목 과제 기한 확인
□ 프로젝트 일정과 비교

각 주별 계획:
11월 15-20: 기획 100% (자유)
11월 22-28: 개발 70% (고사 준비 시작)
11월 29~12월 5: 개발 50% (고사 주간)
12월 6-12: 개발 100% (고사 끝남)
12월 13-19: 배포 100% (다른 과제 끝남)
12월 20-26: 발표 100%
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ 다른 과목 과제 기한 확인
□ 예상 진행 시간 vs 실제 시간
□ 일정 조정 필요한지 확인
```

**응급 대응 (Contingency)**:
```
만약 1-2주 진행 불가하면:
1. Should 기능 중 1-2개 제거 (→ Won't)
2. 테스트 범위 축소 (필수 시나리오만)
3. 문서화 단순화
4. 배포는 꼭 진행 (최소한의 배포 버전)
```

**일정 버퍼**: **+7일** (전체 일정에 이미 포함)

### 위험 4️⃣: Stripe 결제 통합 복잡도

**카테고리**: 기술 (Technology)  
**영향도**: 🟡 **중간** (일정 2-3일 지연)  
**발생확률**: 🟡 **중간** (처음 결제 시스템)  
**총 위험도**: **중간**

#### 설명

Stripe 결제 시스템:
- API 문서 길고 복잡함
- 테스트 환경 설정 어려움
- 보안 (PCI DSS 준수)
- 환율, 세금 처리

#### 영향

```
최악의 경우:
- 결제 실패 로직 구현 못함
- 테스트 카드로 테스트 안 됨
- 결제 후 데이터 불일치
- 13주차 막판에 발견
- 일정 3-4일 지연
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
12주차 중반 (13주차 시작 전):
□ Stripe Flutter 패키지 검토
□ 공식 튜토리얼 1-2개 완료
□ 테스트 환경 설정 (Stripe Test Mode)
□ 테스트 카드로 완전 사이클 테스트
□ 보안 규칙 검토 (API Key 관리)

13주차 초반:
□ 테스트 코드 작성
□ 결제 실패 시나리오 테스트
□ 환율 처리 확인
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ 결제 완료율
□ 결제 실패 로그
□ 보안 이슈 없는지 확인
```

**응급 대응 (Contingency)**:
```
만약 Stripe 통합 어려우면:
1. 결제 → 찜하기로 축소 (예약은 찜 기능만)
2. 기초 결제만 구현 (환불, 세금 제거)
3. Stripe 대신 카카오페이 (다른 SDK)
4. 최악: 앱 내 결제 X, 외부 링크만 (기능 없음)
```

**일정 버퍼**: **+3일** (13주차에 추가)

### 위험 6️⃣: 데이터 손실 및 보안

**카테고리**: 외부 의존 (External Dependency)  
**영향도**: 🔴 **매우 높음** (프로젝트 롤백)  
**발생확률**: 🟢 **낮음** (Firebase 관리형)  
**총 위험도**: **중간**

#### 설명

Firebase 데이터 문제:
- 실수로 데이터 삭제 (스크립트 오류)
- 보안 규칙 설정 오류 → 데이터 공개
- API 키 유출 → 악의적 사용

#### 영향

```
만약 발생하면:
- 모든 카페, 예약, 리뷰 데이터 손실
- 프로젝트 처음부터 다시 시작
- 일정 3-4일 손실
- 데이터 유출시 보안 이슈
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
10-11주차:
□ Firebase 백업 정책 수립
□ 일일 자동 백업 설정
□ Firestore 보안 규칙 검토

12주차부터:
□ 데이터 삽입 스크립트 검증 (DRY RUN 먼저)
□ API Key 환경변수로 관리 (하드코딩 X)
□ 프로덕션 DB 접근 권한 제한

배포 전:
□ 보안 규칙 최종 검토
□ API Key 노출 여부 확인 (GitHub 검색)
□ 백업 완료 확인
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ Firebase 활동 로그 확인
□ 비정상 API 호출 없는지 확인
□ 백업 파일 생성됨 확인
```

**응급 대응 (Contingency)**:
```
만약 데이터 손실:
1. Firebase Backup 복구 (몇 분)
2. 없으면 초기 데이터 50개 재입력 (2-3시간)

만약 API Key 유출:
1. Firebase Console에서 즉시 Key 비활성화
2. 새 Key 발급
3. GitHub 커밋 히스토리에서 Key 제거
```

**일정 버퍼**: **+1일** (응급 대비)

### 위험 7️⃣: AI 의존도

**카테고리**: AI 의존 (AI Dependency)  
**영향도**: 🟡 **중간** (기능 지연)  
**발생확률**: 🟡 **중간** (이번 프로젝트 전부 AI 사용)  
**총 위험도**: **중간**

#### 설명

이 프로젝트 전부 AI로 생성:
- 기획 문서 (Claude)
- 코드 스켈레톤 (Claude)
- 테스트 코드 (Claude)

문제:
- AI가 생성한 코드 버그 가능성
- 본인이 로직 이해 못할 수 있음
- AI가 만든 구조가 프로젝트에 맞지 않을 수 있음

#### 영향

```
최악의 경우:
- AI 코드에 버그 있음 → 디버깅 못함
- 요구사항과 맞지 않은 아키텍처
- 코드 수정 불가능 (이해 못함)
- 전면 재작성 필요
- 일정 5-7일 손실
```

#### 대응 전략

**사전 예방 (Proactive)**:
```
10-11주차:
□ 생성된 문서 100% 이해 & 검토
□ 논리 맞는지 확인
□ 아키텍처 다이어그램 재검토
□ 요구사항과 매핑 확인

11주차부터:
□ AI 코드 받은 후 꼭 읽기 (그냥 복붙 X)
□ 이해 못한 부분 AI에 다시 물어보기
□ 테스트 코드 먼저 읽기 (테스트가 기능 설명)
□ 작은 부분부터 직접 구현해보기

매주:
□ 코드 이해도 자가진단
□ "이 함수가 왜 필요한가?" 설명 가능?
□ 수정 필요시 직접 수정 가능?
```

**위험 감시 (Monitoring)**:
```
매주 체크:
□ 코드 읽을 수 있는가?
□ 버그 찾을 수 있는가?
□ 수정할 수 있는가?
```

**응급 대응 (Contingency)**:
```
만약 AI 코드를 이해 못하면:
1. AI에 "이 코드가 무엇을 하는가" 다시 물어보기
2. 테스트 코드 읽으며 이해하기
3. 필요하면 직접 재작성하기
4. 최악: 다른 방식으로 구현 (시간 더 걸림)
```

**일정 버퍼**: **+3일** (학습 및 재작성)
