# ADR-0002: Firebase 선택

**상태**: 승인됨 ✅  
**작성자**: 팀 리더  

---

## 컨텍스트

모바일 앱의 **백엔드 인프라**를 결정해야 했다.

### 요구사항
- **빠른 구축**: 인프라 설정에 시간 낭비 금지
- **실시간 동기화**: 예약 정보 실시간 업데이트
- **무료**: 초기 스타트업 비용 최소화
- **확장성**: 사용자 증가에 대응

## 의사 결정

**선택**: **Google Firebase** 사용

## 근거

### ✅ Firebase 장점

| 컴포넌트 | 기능 | 이점 |
|---------|------|------|
| **Auth** | 인증 | JWT 토큰 관리 자동화 |
| **Firestore** | NoSQL DB | 실시간 동기화, 자동 인덱싱 |
| **Storage** | 파일 저장 | 이미지 저장 간편 |
| **Hosting** | 배포 | CDN 자동 포함 |

### ❌ 대안 검토

#### AWS (EC2 + RDS + S3)
```
장점: 완전한 제어, 무제한 확장
단점: 설정 복잡 (Terraform), 운영 부담, 비용
비교: Firebase 대비 3배 학습 곡선
결론: 기간 제약에 부적합
```

#### Google Cloud (GCP)
```
장점: Firebase와 동일 생태계
단점: Firebase보다 복잡, 비용 높음
결론: Firebase 사용으로 통일
```

#### 자체 서버 구축
```
장점: 완전 제어 가능
단점: 개발 시간 3배 증가, 운영 복잡
결론: 14주 내 불가능
```

## 기술 스택

```
Authentication:
  ├── Firebase Auth (자동)
  └── JWT Token (24시간 유효)

Database:
  ├── Cloud Firestore
  ├── Real-time Sync
  └── 자동 백업

Storage:
  ├── Cloud Storage
  └── 이미지 저장

Hosting:
  ├── Firebase Hosting (선택사항)
  └── CDN 포함
```

---

## 보안 설계

### Firebase Security Rules

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // 사용자는 자신의 데이터만 접근 가능
    match /users/{userId} {
      allow read, write: if request.auth.uid == userId;
    }

    // 예약 데이터
    match /reservations/{reservationId} {
      allow read: if request.auth.uid == resource.data.userId;
      allow create: if request.auth.uid != null;
    }
  }
}
```
