# ADR-0003: Layered Architecture 선택

**상태**: 승인됨 ✅  
**작성자**: 팀 리더  

---

## 컨텍스트

**아키텍처 패턴**을 선택하여 코드 구조를 정의해야 했다.

### 요구사항
- 계층 간 낮은 결합도
- 각 계층 독립적 테스트 가능
- 장기 유지보수 용이
- 팀 규모에 적합한 구조

---

## 의사 결정

**선택**: **4계층 Layered Architecture**

```
Presentation Layer (UI)
    ↓
Application Layer (State)
    ↓
Domain Layer (Logic)
    ↓
Data Layer (API/DB)
```

---

## 근거

### ✅ 4계층 구조 장점

| 계층 | 책임 | 독립성 |
|------|------|--------|
| **Presentation** | UI 렌더링 | ✓ 높음 |
| **Application** | 상태 관리 | ✓ 높음 |
| **Domain** | 비즈니스 로직 | ✓ 최고 |
| **Data** | API/DB 접근 | ✓ 높음 |

### ❌ 대안 검토

#### 1. MVC 패턴
```
문제점:
  - Controller가 과도하게 비대화
  - 테스트 어려움
  - 유지보수 복잡
결론: 현대 모바일에 부적합
```

#### 2. MVVM + Clean Architecture
```
장점: DDD 패턴 적용
단점: 초기 설정 복잡 (3명 팀에 과함)
결론: 조금 더 단순한 구조 선택
```

#### 3. BLOC 패턴
```
장점: 반응형 프로그래밍
단점: 학습 곡선 높음
결론: Layered + Provider로 충분
```

## 아키텍처 상세

### 1️⃣ Presentation Layer

```dart
lib/presentation/
├── screens/        # UI 화면
├── widgets/        # 재사용 위젯
└── providers/      # 상태 관리
```

**책임**: UI 렌더링 및 사용자 입력 처리만

### 2️⃣ Application Layer

```dart
lib/application/
└── providers/      # Provider 상태 관리
```

**책임**: State 관리, UseCase 호출

### 3️⃣ Domain Layer (비즈니스 로직)

```dart
lib/domain/
├── entities/       # 도메인 모델
├── repositories/   # 인터페이스
└── usecases/       # 비즈니스 로직
```

**책임**: 프레임워크 무관 순수 로직

### 4️⃣ Data Layer

```dart
lib/data/
├── datasources/    # API/DB 호출
├── models/         # 데이터 모델
└── repositories/   # 구현체
```

**책임**: 외부 데이터 접근

---

## 의존성 규칙

```
✓ Presentation → Application
✓ Application → Domain
✓ Domain → Data
✗ Data → Domain (금지!)
✗ 역방향 의존성 (금지!)
```

---

## 테스트 전략

```
Domain Layer (70%)
  └─ UseCase 단위 테스트
  
Data Layer (20%)
  └─ Repository 통합 테스트
  
Presentation Layer (10%)
  └─ Widget UI 테스트
```

---

## 확장성

```
추가 기능 시:
1. Domain 계층에 UseCase 추가
2. Data 계층에 Repository 구현
3. Application 계층에 Provider 추가
4. Presentation 계층에 UI 추가

→ 최소 영향으로 확장 가능
```
